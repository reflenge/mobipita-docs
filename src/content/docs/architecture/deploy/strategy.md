## 概要

MobiPitaでは、Master AppとCustomer Appの2つのアプリケーションをデプロイする必要があります。

開発速度とコスト効率を両立するデプロイ戦略を検討しています。

## デプロイ環境候補の比較

### Vercel (有力候補)

**特徴:**
- Next.jsに最適化されたホスティング
- ゼロインフラ構築
- 自動デプロイとプレビュー環境

**メリット:**
- 開発速度を最大化
- ゼロインフラ構築
- 自動デプロイとプレビュー環境
- ランニングコストを抑える（初期段階）

**デメリット:**
- コストが高くなる可能性（成長に伴い）
- ベンダーロックインのリスク
- エンタープライズ要件への対応が限定的

**マルチテナント対応:**
- サブドメインによるテナント分離が可能
- 各テナントごとの独立した環境を構築可能

### AWS

**特徴:**
- 業界標準のクラウドインフラ
- 高いスケーラビリティ
- 豊富なサービスと統合

**メリット:**
- コスト最適化が可能
- エンタープライズ要件への対応
- ベンダーロックイン回避
- 高いスケーラビリティ

**デメリット:**
- 初期設定が複雑
- 運用・保守の責任が大きい
- 学習コストが高い

**マルチテナント対応:**
- App Runner、ECS、Lambdaなど複数の選択肢
- サブドメインによるテナント分離が可能

## 推奨戦略

**初期段階**: Vercel

- 開発速度を最大化
- ゼロインフラ構築
- ランニングコストを抑える

**移行先**: AWS

- PMF後のコスト最適化
- エンタープライズ要件への対応
- ベンダーロックイン回避の実現

## 移行ロードマップ

詳細は[インフラ選定](/research/infrastructure-selection)の「AWS移行ロードマップ」を参照してください。

### Phase 1: アプリケーションのコンテナ化

Next.jsアプリケーションをDockerコンテナとして動作するようにします。

### Phase 2: AWS環境の構築

Vercelの役割（Compute）のみをAWSに移します。

### Phase 3: インフラの切り替え

DNS設定を変更し、トラフィックをVercelからAWSへ向けます。

### Phase 4: 完全移行

コストや要件に応じて、周辺サービスもAWSネイティブに寄せます。

:::tip[実装前の検討事項]
以下の点は実装前に確定する必要があります：
- デプロイ環境の最終選定（Vercel / AWS）
- 移行タイミングの判断基準
- マルチテナント対応の実装方法
- コスト見積もり

これらは実装フェーズで詳細化される予定です。
:::
