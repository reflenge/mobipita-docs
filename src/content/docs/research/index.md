## 概要

本セクションでは、技術選定やインフラに関する調査結果をまとめています。

プロジェクトの技術スタック選定にあたり、様々な技術オプションを調査し、比較検討した結果を記録しています。

### 調査の目的

- **技術選定の根拠**: なぜその技術を選んだかの理由を明確化
- **比較検討**: 複数のオプションを比較し、最適な選択を記録
- **移行戦略**: 将来的な移行計画を明確化

## 主要なドキュメント

### インフラ選定

**ファイル**: [インフラ選定](/research/infrastructure-selection)

B2B2Cアプリケーションにおけるインフラ選定と移行戦略について解説します。

#### 主要な内容

- **主要4大クラウド比較**: AWS、GCP、Azure、OCIの比較と選定指針
  - **AWS**: 移行先の最終目標（PMF後）
  - **GCP**: 今回は見送り（データ分析特化ではない）
  - **Azure**: 今回は見送り（Web系ではAWSが主流）
  - **OCI**: 部分的採用もアリ（将来的に動画や大量データを扱う場合）
- **初期構成案**: Vercel + Clerk + R2 (Modern PaaS)
  - 開発スピード（Time to Market）を最大化
  - ランニングコストを抑える
  - ゼロインフラ構築
- **ベンダーロックイン回避**: AWS移行を前提とした技術選定基準
  - コンピューティング: Node.js環境を使用
  - ストレージ: AWS SDK v3を使用（R2はS3互換）
  - データベース: Neon/Supabase + Prisma/Drizzle ORM
  - 非同期処理: Inngest, Trigger.dev, またはAPI Route
- **AWS移行ロードマップ**: Exit Strategyの計画
  - Phase 1: アプリケーションのコンテナ化
  - Phase 2: AWS環境の構築（Compute移行）
  - Phase 3: インフラの切り替え（DNS Switch）
  - Phase 4: 完全移行（必要な場合のみ）

[詳細を見る →](/research/infrastructure-selection)

## 調査の背景

MobiPitaプロジェクトでは、以下の観点から技術選定を行っています：

1. **開発速度**: Time to Marketを最大化
2. **コスト効率**: 初期コストを抑えつつ、成長に合わせてスケール可能
3. **ベンダーロックイン回避**: 将来的な移行を考慮した技術選定
4. **セキュリティ**: マルチテナントアプリケーションに適した構成

## 推奨構成

**初期段階**: Vercel + Clerk + R2 (Modern PaaS)

- 開発スピードを最大化
- ランニングコストを抑える
- ゼロインフラ構築

**移行先**: AWS（PMF後）

- コスト最適化と統制
- エンタープライズ要件への対応
- ベンダーロックイン回避の実現

詳細は[インフラ選定](/research/infrastructure-selection)を参照してください。

## 調査方針

本プロジェクトの調査では、以下の方針に従っています：

1. **実用性重視**: 理論だけでなく、実運用を考慮した選定
2. **段階的移行**: 初期は開発速度を優先し、成長に合わせて移行
3. **標準技術**: ベンダー固有の機能に依存しない設計
4. **ドキュメント化**: 選定理由と移行戦略を明確に記録
